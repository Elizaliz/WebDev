Standards Background

One of the reasons HTML has done so well, is that it is quite forgiving and has the policy to fail gracefully rather than not work at all if a browser comes across something it doesn't quite understand. With the increased use of XML, a more formal, less forgiving version of HTML was created by the W3C group, XHTML. XHTML attempts to support every feature of HTML 4.01 but uses the more stringent rules of XML. We've already seen how in HTML some tags don't require end tags, but in XHTML, all tags require end tags. This applies to even tags like <br> which must be used in the <br /> form for XHTML compliance.

It is true that IE 8 will operate in standards mode, which helps migrate users away from Microsoft HTML extensions/variation. However, in a recent (2009) report by browser vendor Opera Software ASA showed that the average page on the WEb strays from WC3 page markup specifications 47 times. Overall, only around 4% of the 3.5 million Web pages Opera searched were fully standards compliant when tested against the W3C validation suite.

Why such poor numbers. Several factors:

  • Olders sites that haven't been updated
  • Developers don't use standards
  • Different browsers still interpret some standards differently
  • Not all of the W3C standard features are implemented
  • Some sites still block users with "non approved" browsers.
  • Some specifications still are ambiguous, allowing different vendors to come up with completely different solutions.

XHTML Background

XHTML was the first step toward a modular and extensive web based on XML. XHTML is a "Reformulation" of HTML 4.0 that can operate in HTML conforming browsers

  • Supports all markup tags and attributes from HTML 4.0
  • Elements must be properly nested – (i.e., well formed)
  • All elements must be closed
  • Tag names in lower case
  • Attribute names must be in lower case
  • Attributes must have arguments
  • Attribute values must be quoted

Why bother with XHTML? Since it is now well-formed XML, it can be viewed, transformed and validated with XML tools. It is stricter and cleaner than HTML4.0, and is more readily applicable to devices other than computes (mobile phones, pdas, etc).

Since XHTML is more formal, there are additional restrictions over HTML. First, case matters, and all tags shall be in lower case. Second, in HTML quotes for an attribute value are optional if it is just plain text. In XHTML double or single quotes are required for all attribute values.

An example of XHTML is as follows:

<?xml version="1.0" encoding="UTF-8"?>

<!DOCTYPE html PUBLIC "-//W3C//DTD XHTML 1.0 Transitional//EN"

"http://www.w3.org/TR/xhtml1/DTD/xhtml1-transitional.dtd">

<html xmlns="http://www.w3.org/1999/xhtml">

<head><title>Ajax: Simple Message</title>

</head>

<body>

...

</body></html>

Notice the different DOCTYPE value, which specifies the XHTML dtd.

Validation of XHTML

You can check the syntax using a formal validator, such as:

  • http://validator.w3.org
  • http:/www.htmlhelp.com/tools/validator

The version of XHTML is based on the DOCTYPE shown in the example above.

Future of XHTML

Even though XHTML is considered to be superior to HTML by web designers, there are several major issues with it. First, IE (through version 7) doesn't support it, which is a significant issue. It turns out that there wasn't supposed to be an HTML 5 version, that instead the XHTML standard would replace the HTML standard. The problem is, the industry isn't really supporting XHTML and there has been enough pressure that the WWW Consortium is now working on HTML 5 as the next major web upgrade. XHTML is still supported, for example, HTML 5 can be described by XHTML. The big issue is that it doesn't look like XHTML will be forced on browsers, and consequently, probably won't the the de-facto standard.

So, if it isn't required, why use it? The biggest reason is that XHTML can be put through EXtensible Stylesheet Lanugage (XSLT) which allows you to transform one XML document into another. You can't do that with regular HTML. This would allow you to have a single XML document that would describe a web page, and then use different XSLT translators to customize the XHTML to the browsing platform.
